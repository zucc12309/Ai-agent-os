# DevOps & Runtime Review

Scope: Dockerfiles, docker-compose, environment/config, migrations/init DB, README setup, dependency versions, startup order, and health checks.

No source code was changed for this review.

## Executive Summary

The local compose path is close to bootable, but the repository is not ship-ready for a sensitive enterprise deployment yet. The biggest issues are a plaintext demo API key being printed during startup, the frontend bundle being baked to `localhost`, no migration workflow, and runtime surfaces that can erase audit history or reach unintended internal hosts.

The codebase does have a few solid runtime foundations: PostgreSQL already has a healthcheck in compose, API keys are hashed with a pepper, and the config parser handles comma-separated CORS origins. Those are good starts, but they are not enough to make this safe to ship.

## Local Run Blockers

- No fatal syntax or import blocker was obvious from inspection, and the compose file does start PostgreSQL before the backend. The first-boot caveat is that the only operator key is emitted to stdout during seeding, so local login depends on reading container logs. Evidence: [docker-compose.yml](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/docker-compose.yml#L12), [backend/app/main.py](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/backend/app/main.py#L27), [backend/app/seed.py](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/backend/app/seed.py#L164).

## Deployment Blockers

- **[P0] Startup seeding leaks a plaintext admin key and is always enabled.** Evidence: [backend/app/main.py](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/backend/app/main.py#L27) seeds the database during app lifespan, [backend/app/seed.py](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/backend/app/seed.py#L164) generates and prints `Demo agent API key`, and [README.md](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/README.md#L96) tells operators to copy the key from logs. Why it matters: the first boot places a fully privileged operator credential into container logs and any log aggregator. Exact fix: gate seeding behind an explicit `DEMO_SEED_ENABLED` flag, default it off for production, stop printing the key outside a local bootstrap command, and make production startup fail closed if demo seeding is disabled.

- **[P0] The frontend bundle is compiled against `localhost`.** Evidence: [frontend/lib/api.ts](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/frontend/lib/api.ts#L1) defaults the API origin to `http://localhost:8000`, [frontend/Dockerfile](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/frontend/Dockerfile#L9) bakes that value in at build time, and [docker-compose.yml](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/docker-compose.yml#L34) repeats the same localhost origin. Why it matters: any deployment accessed from another machine will have browser requests sent to the user’s own localhost, so dashboard API calls fail. Exact fix: move API traffic behind the same origin via a reverse proxy or Next rewrite, or inject the public API origin at deploy time instead of compiling `NEXT_PUBLIC_API_BASE_URL` to `localhost`.

- **[P0] Internal API execution can become an SSRF primitive.** Evidence: [backend/app/connectors/internal_api_connector.py](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/backend/app/connectors/internal_api_connector.py#L19) joins a user-controlled `path` onto `INTERNAL_API_BASE_URL` and forwards arbitrary headers and methods. Why it matters: absolute URLs or scheme-relative paths can escape the intended base host, and headers can carry credentials to the wrong destination. Exact fix: reject absolute URLs and any path containing scheme/netloc, enforce a strict allowlist of methods and paths, and only permit egress to pre-approved hosts.

- **[P0] The safe SQL connector still runs arbitrary SQL against the primary app database role.** Evidence: [backend/app/connectors/postgres_connector.py](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/backend/app/connectors/postgres_connector.py#L36) executes text SQL supplied by the caller, and [backend/app/database.py](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/backend/app/database.py#L17) creates the app engine with the main database credentials. Why it matters: agents can read any table the application role can access, including audit and agent tables, and can also DoS the database with expensive reads. Exact fix: move the connector to a dedicated read-only DB role with schema/table grants, or replace raw SQL with whitelisted query templates and AST-level validation.

- **[P1] Audit history is not retention-safe.** Evidence: [backend/app/models/audit_log.py](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/backend/app/models/audit_log.py#L13), [backend/app/models/tool.py](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/backend/app/models/tool.py#L42), [backend/app/models/approval.py](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/backend/app/models/approval.py#L13), and [backend/app/models/agent.py](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/backend/app/models/agent.py#L13) all use delete-oriented FK/cascade behavior that can remove approval or audit rows when parent records disappear. Why it matters: deleting a tool or agent can erase the very records needed for incident review and compliance. Exact fix: remove delete-orphan cascades from audit/approval relationships, prefer `RESTRICT` or `SET NULL` with immutable snapshots, and never make audit rows disappear just because a parent record was deleted.

- **[P1] There is no Alembic migration path; schema creation happens inside the app process.** Evidence: [backend/app/database.py](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/backend/app/database.py#L34) calls `Base.metadata.create_all`, and [backend/app/main.py](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/backend/app/main.py#L27) runs that during lifespan before serving traffic. Why it matters: you cannot safely evolve the schema, run repeatable upgrades, or deploy multiple replicas without startup races. Exact fix: add Alembic, generate versioned migrations, and move schema changes out of the request-serving process.

- **[P1] Backend and frontend readiness are not modeled.** Evidence: [docker-compose.yml](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/docker-compose.yml#L18) only defines a healthcheck for PostgreSQL, while backend and frontend have no healthcheck and the frontend only depends on backend by container start order. Why it matters: the UI can come up before the backend is actually ready, and orchestrators cannot make a real readiness decision. Exact fix: add `HEALTHCHECK` to the backend and frontend images or compose services, expose a lightweight backend readiness endpoint, and gate frontend startup on backend health.

- **[P1] Approval execution is not atomic.** Evidence: [backend/app/routers/approvals.py](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/backend/app/routers/approvals.py#L51) marks an approval as approved and then calls `execute_tool`, while [backend/app/services/approval_service.py](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/backend/app/services/approval_service.py#L107) commits each state change separately. Why it matters: concurrent approvers or a crash between steps can leave approval, execution, and audit state out of sync or allow duplicate execution. Exact fix: lock the approval row in a single transaction, re-check `pending` after the lock, and persist approval plus execution outcomes atomically.

- **[P1] Audit logs capture raw payloads without redaction or size limits.** Evidence: [backend/app/services/audit_service.py](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/backend/app/services/audit_service.py#L13) stores full input and output payloads, [backend/app/services/execution_service.py](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/backend/app/services/execution_service.py#L118) passes through connector results and error strings, [backend/app/routers/audit.py](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/backend/app/routers/audit.py#L15) returns them to the UI, and [frontend/app/audit/page.tsx](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/frontend/app/audit/page.tsx#L66) renders them directly. Why it matters: PII, tokens, and internal data can end up in the audit database and browser UI. Exact fix: add a redaction layer, cap stored payload size, store only safe summaries by default, and explicitly whitelist fields that are allowed to appear in the UI.

- **[P2] Dependency builds are not reproducible.** Evidence: [backend/requirements.txt](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/backend/requirements.txt#L1) uses open version ranges, [frontend/package.json](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/frontend/package.json#L10) uses caret ranges, and [frontend/Dockerfile](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/frontend/Dockerfile#L4) runs `npm install` without a committed lockfile. Why it matters: a rebuild can pick up different dependency versions and change runtime behavior without a code change. Exact fix: pin versions, add lockfiles, and use `npm ci` in the frontend image.

- **[P2] The health endpoint is too expensive for frequent probing.** Evidence: [backend/app/main.py](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/backend/app/main.py#L58) counts agents, tools, approvals, and audit logs on every `/health` request. Why it matters: repeated health probes or status page refreshes will scan the largest table in the system as audit volume grows. Exact fix: split liveness from readiness and keep the public health response to a cheap DB ping plus a small number of cached counters.

- **[P2] The compose and env story is local-only and should be marked as such.** Evidence: [docker-compose.yml](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/docker-compose.yml#L4) hardcodes local credentials and localhost origins, and [.env.example](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/.env.example#L1) repeats local defaults without production variants. Why it matters: someone can treat the example compose file as deployable and end up with weak defaults, broken cross-origin access, and the wrong frontend API target. Exact fix: add production examples for the public frontend origin, CORS origins, DB credentials, API key pepper, and a non-local API base URL, and fail fast if placeholders are still in use outside development.

## Missing Configs

- `DEMO_SEED_ENABLED` exists in [backend/app/config.py](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/backend/app/config.py#L36) but is not wired into startup and is not documented in [.env.example](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/.env.example#L1). Add the env var, default it off for production, and gate the seed path on it.

- `MCP_MOUNT_PATH` exists in [backend/app/config.py](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/backend/app/config.py#L32) but [backend/app/main.py](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/backend/app/main.py#L55) still mounts `/mcp` literally. Either wire the setting into the mount path or remove the config field so the runtime and docs cannot drift.

- `ENVIRONMENT` and `LOG_LEVEL` are present as code defaults in [backend/app/config.py](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/backend/app/config.py#L18) but are not documented in [.env.example](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/.env.example#L1). Add them so production deployments can set environment mode and logging explicitly.

- The example env file only documents local endpoints. Add production variants for `NEXT_PUBLIC_API_BASE_URL`, `CORS_ORIGINS`, `DATABASE_URL`, `API_KEY_PEPPER`, and the Postgres credentials in [.env.example](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/.env.example#L1) so the repo does not look production-ready while still encoding localhost-only defaults.

## What Is Already Good

- The PostgreSQL service already has a healthcheck in [docker-compose.yml](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/docker-compose.yml#L12), which is the right direction for startup gating.

- API keys are hashed with a pepper in [backend/app/services/auth_service.py](/Users/priyanshupatel/Documents/GitHub/ai%20operating%20system/backend/app/services/auth_service.py#L22), which is the correct baseline for MVP auth. The remaining issue is the default placeholder pepper and the plaintext demo key leak during seeding.

## Recommended Implementation Order

1. Remove the startup secret leak and make demo seeding opt-in for development only.
2. Decouple the frontend from a baked-in localhost API origin.
3. Add Alembic migrations and stop relying on `create_all` during app startup.
4. Lock down connector execution, especially internal API egress and raw SQL.
5. Make audit records immutable and add redaction or size limits before shipping to enterprise users.
6. Add readiness/health checks and pin dependency versions so the stack is reproducible and observable.
